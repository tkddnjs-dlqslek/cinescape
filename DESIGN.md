## Problem Statement
콜미바이유어네임, 비포 선라이즈, 미드나잇 인 파리 같은 영화를 보면 그 장소를 직접 가보고 싶어진다.
하지만 "어느 장면이 정확히 어디서 찍혔는지 + 거기 가면 지금은 어떤 모습인지 + 한 번 여행에서 가까운 촬영지를 묶어 도는 동선"을
감각적으로 탐색하게 해주는 서비스가 없다. 시네필을 위한 "촬영지 여행 도우미"를 세계지도 위에서 만든다.

## What Makes This Cool (히어로 = "와" 포인트)
**핵심 delight: Then/Now 장면 오버레이.** 핀을 클릭하면 영화 스틸컷과 그 장소의 현재 모습이 슬라이더로 겹쳐진다.
"그 벤치가, 그 광장이 아직 거기 있다"는 환희. 이게 v1이 책임지고 완성해야 할 단 하나의 경험.

기능 우선순위 (사용자 확정):
1. **검색 → 줌인 → Then/Now** (최우선, 히어로)
2. 가까운 촬영지 3곳 동선 추천 (한 번 여행으로 여러 영화)
3. AI/반자동 데이터 추출 파이프라인 (확장)
4. "이 도시의 영화 지도" 밀도 뷰

### 경쟁 풍경과 차별점 (레퍼런스 조사 결과)
- **FilmingMap** (filmingmap.com): 3D 글로브 + 1,530편. 비전상 가장 가깝지만 **유틸리티 지도라 못생겼고 Then/Now·스틸·동선이 전부 없음.** → 정확히 우리가 칠 갭.
- **SetJetters** (setjetters.com): 모바일 앱 + AR ShotSync(내 사진 vs 영화 샷). Then/Now 강하지만 앱·소셜 중심, 감각적 웹 지도 아님.
- **ReelStreets** (reelstreets.com): Then/Now 캡처 96,000장의 압도적 아카이브지만 지도가 아닌 못생긴 DB.
- **CineMapper / MovieMaps**: 텍스트·리스트 위주.
- **EUREKA**: 핀 박힌 글로브는 이미 존재한다(FilmingMap). 진짜 해자는 **시네필 큐레이션 + Then/Now 오버레이 + 일관된 영화적 무드** 세 가지의 결합.
  Awwwards 수상작(San Rita Project, Cartier 2025, ERA Real Estate)이 증명: "기능 수보다 일관된 분위기(atmosphere)가 이긴다."

## Constraints
- 포트폴리오/비상업 데모. 1인 빌드.
- 스틸컷은 영화사 저작권 → 비상업·데모 전제로 진행, 상업 런칭 시 라이선스는 별도 과제.
- 실제 고감도 디자인은 빌드 단계에서 (정적 목업이 아닌 react-globe.gl 실물에서 감도가 나옴).

## Premises (확정)
1. **히어로의 진짜 난이도는 "정확한 vantage 좌표 + 그 프레임 스틸컷" 페어링이다.** "이 카페가 나왔다"가 아니라 "카메라가 여기서 이 각도로". 핀 1개당 데이터가 일반 지도보다 무겁다. → 동의
2. **스틸컷은 저작권 자산.** 비상업 데모 전제로 명시. → 동의
3. **v1 시드 = 손큐레이션 명작 10~20편.** ~~유럽 1~2개 지역 집중~~ → **수정: 전 세계(권역 무관).** 단, "근처 3곳 동선"과 밀도감을 위해 자연스럽게 뭉치는 도시(파리·로마·뉴욕 등)에 시드를 클러스터링한다. → 동의(수정 반영)
4. **프론트 중심 지도 앱.** 진짜 지도/글로브 라이브러리 + 정적 JSON/SQLite면 v1엔 무거운 백엔드 불필요. 배포 = Vercel. → 동의

## Approaches Considered
### Approach A — Lean Static Map (최소 실행, S/Low)
정적 JSON 데이터셋(~15편) + 지도 라이브러리 + Then/Now 슬라이더. 백엔드 없음. 히어로를 가장 빨리 완성. 데이터 확장은 수동.

### Approach B — Structured Data Platform (이상적, L/Med)
정형 스키마(films/locations/scenes) + 인제스트 스크립트 + AI 추출/동선을 1급 시민으로. 확장성 최고지만 히어로 완성이 늦고 v1엔 과설계.

### Approach C — Cinematic Globe (창의적, M~L/Med~High)
회전 글로브를 진입으로, 영화 선택 시 카메라가 예고편처럼 flyTo → Then/Now reveal. 가장 기억에 남지만 폴리시 리스크.

## Recommended Approach — 하이브리드 (확정)
**A를 척추로, C의 글로브 인트로 + B의 깔끔한 scene 스키마를 이식.**
- 이유: 히어로(①)를 가장 빨리 책임지고 완성하면서, 진입의 "와"(C)와 확장성 토대(B)를 저비용으로 챙긴다.
- 포트폴리오는 "끝까지 다듬은 한 경험"이 최고의 증거.

### 시각 방향 (확정): Direction C — Midnight Glass
심야 코발트 그라데이션 · 3D 글로브 · 글래스모피즘 패널(블러) · 시안/바이올렛 액센트 · Instrument Serif 타이포.
Awwwards 수상 트렌드(다크·시네마틱)에 가장 부합. **고감도 디자인은 빌드 단계에서** react-globe.gl 실물 + 실제 스틸 + 마이크로 인터랙션으로 구현.
(무드 목업 3종 보관: scratchpad/mockups/ — A_cinema_noir, B_vintage_atlas, C_midnight_glass. C 채택.)

### 기술 스택 (제안)
- **프레임워크**: Next.js (App Router) + TypeScript
- **글로브/지도**: react-globe.gl (ThreeJS) — flyTo, 대기광, 핀 레이어. (필요 시 MapLibre 평면 폴백)
- **데이터**: v1은 정적 JSON / SQLite. scene 스키마: `{ film, year, director, tmdb_id, scenes: [{ name, note, coord:{lat,lng}, still_url, now_url, vantage_bearing }] }`
- **배포**: Vercel

### 데이터 소스 (스케일 경로 — 반자동)
- **Wikidata WikiProject Filming Locations**: `P915(filming location)` + `P625(coordinate location)` → SPARQL 한 방으로 좌표 박힌 촬영지 쿼리. 무료.
- **TMDB API**: 영화 메타데이터·포스터·스틸. Wikidata와 `TMDB movie ID(P4947)`로 연결.
- **LLM**: "어느 장면인지" 보강(자막/위키/기사 파싱)에만 한정 사용.
- → ④AI추출 = "Wikidata SPARQL로 좌표 + TMDB로 스틸 + LLM으로 장면 태깅"으로 분해되어 현실화.

## Open Questions
- vantage 좌표(카메라 각도/위치)의 정확도를 어디까지 추구할지 — v1은 "장소 좌표 + 대략 bearing"으로 시작, 명작은 손보정.
- "now" 현장 사진 소스: 직접 촬영 불가 → Google Street View / Mapillary / Wikimedia Commons 중 라이선스 검토.
- Letterboxd 리스트 임포트(배포 훅) — v2 검토.

## Success Criteria
- v1: 손큐레이션 10~20편, 글로브에서 검색→flyTo→Then/Now 슬라이더가 끊김 없이 동작, 최소 1개 도시에서 "근처 3곳 동선"이 실제로 묶임.
- 채용담당자/친구가 콜미바이유어네임을 검색해 Then/Now를 직접 슬라이드해보고 "와" 하는 순간.
- Vercel 라이브 URL 배포.

## Distribution Plan
Vercel 웹앱 (기존 배포 파이프라인으로 커버). 도메인은 선택. PWA/모바일 반응형 고려.

## Next Steps (dynamic workflow 빌드)
사용자 합의: **디자인 + 데이터 수집은 dynamic workflow로 해결.** 빌드 단계 제안 순서:
1. **데이터 검증 스파이크**: Wikidata SPARQL로 콜미바이유어네임 등 명작의 실제 좌표가 얼마나 나오는지 확인 (가장 먼저 리스크 제거).
2. **시드 데이터셋**: 손큐레이션 10~20편 × scene 스키마 JSON 작성 (클러스터링되는 도시 우선).
3. **글로브 + Then/Now 히어로**: Next.js + react-globe.gl, 검색→flyTo→슬라이더.
4. **근처 3곳 동선**: 좌표 기반 클러스터링 + 동선 카드.
5. **반자동 인제스트 파이프라인**: SPARQL+TMDB+LLM으로 확장.
6. **고감도 디자인 패스**: C 무드로 마이크로 인터랙션·타이포·대기광 폴리시 (`/design-review`).

## The Assignment (다음 행동 한 가지)
**콜미바이유어네임 한 편으로 "골든 패스" 데이터를 손으로 채워봐라.**
- 주요 장면 5~8개의 (장면 설명 + 정확한 위경도 + 영화 스틸 1장 + 현재 사진 1장)을 직접 모아 JSON 한 파일로.
- 이게 되면 전체 제품의 데이터 난이도·스틸 라이선스·vantage 정밀도가 한 번에 검증된다. 이게 v1의 진짜 리스크다.

## What I noticed about how you think
- "1,2,3,4 기능 다 들어갈 거긴 한데, 1번이 제일 중요하지" — 욕심내되 우선순위를 분명히 아는 사람. 이게 빌더의 핵심 근육.
- "일단 1,3으로 기본 DB 구축해서 시연해보고, 그 후에 2,4" — 검증부터 하고 스케일하는 순서 감각이 자연스럽다.
- "래퍼런스 먼저 찾자" / "아직 더 볼래" — 결정 전에 풍경을 충분히 보는 신중함. 덕분에 Wikidata+TMDB 경로와 차별점을 제대로 잡았다.
- "빌드 단계에서 진짜로" — 정적 시안에 시간을 태우지 않고 실물에서 감도를 올리는 판단. 도구의 한계를 아는 선택.
